MySQL

ACID

A: Atomicity（原子性）

事务被视为不可分割的最小单元，事务的所有操作要么全部提交成功，要么全部失败回滚。

回滚可以用undo log来实现，undo log记录着事务所执行的修改操作，在回滚时反向执行这些修改操作即可。

C: consistency（一致性）

数据库在事务执行前后都保持一致性状态。在一致性状态下，所有事务对一个数据的读取结果都是相同的。

I: isolation（隔离性）

如果不存在如下4个隔离级别，就是不同事务写数据没有加锁，会导致数据库的数据出现错误。

4个隔离级别，

  • read uncommitted（读未提交）

    不同事务写数据加锁（排它锁），但是读数据不加锁。因此存在脏读，即读取了其他事务未提交的数据。

  • read committed（读已提交）

    解决了脏读，写数据加锁（排它锁），读数据加锁（共享锁），且规定一个数据不能同时持有排它锁和共享锁（在写的时候不能读），但是能同时持有多个共享锁。会导致不可重复读，即在一个事务期间，对数据的多次读取结果不一致，其原因是多次读取过程中有其他事务修改了该值。

  • repeatable read（可重复读）MySQL默认的隔离级别

    解决了不可重复读，修改了共享锁，在读已提交中，只有在读取数据的时候才会加共享锁，修改为了开始事务的时候就加共享锁，事务提交的时候才释放。但是又会导致幻读，例如有一个事务查询了几列(Row)数据，而另一个事务却在此时插入了新的几列数据，先前的事务在接下来的查询中，就会发现有几列数据是它先前所没有的。

  • serializable（串行化）

    　事务只能一件一件的进行，不能并发进行。解决了幻读，但是带来严重的性能问题。

在读已提交和可重复读下，MySQL 都会创建视图（consistent read view, 一致性读视图），不同的是，读已提交视图是在每个SQL语句开始执行的时候创建的，所以同一个事务内两次相同的查询可能会查到不同的数据；而可重复读是在事务启动的时候创建的，整个事务存在期间都使用该视图。

D: durability（持久性）

一旦事务提交，则其所做的修改将会永远保存到数据库中。即使系统发生崩溃，事务执行的结果也不能丢失。 使用redo log来保证持久性。

MySQL 内存管理之LRU机制

在WAL机制中，InnoDB 内存的一个作用，是保存更新的结果，再配合 redo log，就避免了随机写盘。此外，MySQL Buffer pool的命中率和性能息息相关，为了提高Buffer pool的命中率，MySQL使用LRU管理内存。但是使用基础的LRU会有一个问题，就是执行冷数据的全表扫描的时候，会把所有热数据都替换出去，使内存命中率急剧下降。因此，MySQL对LRU做了改进：将LRU链表按照5:3分成yang区和old区。表头5/8的数据是yang区，之后的是old区。

  • 处于yang区的数据访问和LRU算法相同。

  • 对于old区的数据访问，每次访问是都要判断：1. 如果这个数据页在 LRU 链表中存在的时间超过了 1 秒，就把它移动到链表头部。2. 如果这个数据页在 LRU 链表中存在的时间短于 1 秒，位置保持不变。1秒由MySQL设置。

  • 对于不在链表里的数据的访问，首先淘汰掉链表末尾的数据页 （也是old区末尾的数据区），但是新插入的数据页是放在old区的起始位置。

这样在全表扫描冷数据的时候就不会把yang区的数据替换出去。

MySQL回表优化之Multi-Range Read 优化

MySQL的非聚集索引如果没用到覆盖索引的话会回表，即非聚集索引B+树叶子节点保存的是主键的值，要通过这个值再去主键索引上找数据。这里即使再非聚集索引上使用范围查询是循序查找，取出主键值之后，主键值也会变成随机的，再回表的话就得在主键索引上做随机查询，徒增性能损耗。

MySQL的优化方式是，在非聚集索引上的范围查询取出主键值之后，先把这些主键值缓存在read_rnd_buffer中，如果read_rnd_buffer满了，先对缓存的主键值排序，之后再回主键索引上查找。这样就可以利用B+树顺序查找的优点优化性能。

MySQL之索引下推

我们知道MySQL的最左匹配原则，但是最左匹配在匹配到范围索引的时候（like（非字符串索引的最左 M 个字符），＞，＜，between），就会停止继续向右匹配，如下：

select * from tuser where name like ‘张%’ and age=10 and ismale=1;

表中有联合索引（name, age）。如果没有索引下推，只能从like找到的数据一个个回表，再比对字段值判断是否符合。如果使用了索引下推，就可以在索引遍历过程中，对索引中包含的字段先做判断，直接过滤掉不满足条件的记录，减少回表次数。

Redis

数据结构与对象

Redis SDS空间预分配策略

对SDS修改后，如果SDS的长度小于1MB，则会分配和len长度相等的未使用空间，即设置free = len，并分配2len的空间；如果SDS的长度大于等于1MB，则会分配1MB的未使用空间。
惰性空间释放，即SDS缩短buf内的字符串时不会立即释放空间，而是把未使用的空间记录到free中。

Redis链表

redis的链表使用头插法，使用murmurhash2计算hash

Redis字典

哈希表rehash扩容策略是，ht[1]的size为大于等于ht[0].used * 2的第一个2的幂。当目前没有BGSAVE或者BGREWRITEAOF命令执行，且哈希表的负载因子大于等于1；或者正在执行BGSAVE或者BGREWRITEAOF，且哈希表的负载因子大于等于5时会自动开始哈希表的rehash。

rehash时需要将ht[0]的所有数据rehash到ht[1]，这个过程不是一次性完成的，而是分步多次进行的，渐进式rehash。在渐进式rehash时，所有新添加的k-v都保存到ht[1]中，而查找、删除、更新等操作会在两个表上进行。

Redis跳表

Redis只在两个地方用到了跳表，

  1. 是有序集合键

  2. 是集群节点中用作内部数据结构

每次创建新的跳表节点的时候都会随机生成一个1-32之间的值作为level数组的大小，表示层的高度。

整数集合

整数集合保证不会出现重复的元素

压缩链表

压缩链表是列表键和哈希键的底层实现之一。当列表键只包含少量列表项，并且列表项是小整数值或比较短的字符串时，会使用压缩列表来实现。当哈希键只包含少量键值对，且键和值是小整数值或比较短的字符串时，会使用压缩链表来实现哈希键。

对象

Redis包含5种类型的对象，分别是字符串对象、列表对象、哈希对象、集合对象和有序集合对象。

字符串对象(string)

字符串对象的编码可以是int, raw, embstr。Redis使用对象来表示数据库中的键和值，键总是一个字符串对象。对于字符串对象，如果保存的时是整数值且可以用long保存，那么字符串对象的编码为int；如果字符串对象保存的是字符串值，且长度大于32字节，则使用RAW（SDS）来保存，如果长度小于等于32字节则使用EMBSTR编码的SDS来保存。

列表对象(list)

列表对象的编码可以是ziplist和linkedlist，当满足以下两个条件的时候才会采用ziplist编码：

  1. 列表对象保存的所有字符串元素的长度都小于64个字节
  2. 列表对象保存的元素数量小于512个
哈希对象(hash)

哈希对象的编码可以是ziplist和hashtable，当满足以下两个条件的时候才会采用ziplist编码：

  1. 哈希对象保存的所有键值对的键和值的字符串的长度都小于64个字节
  2. 列表对象保存的键值对数量数量小于512个
集合对象(set)

集合对象的编码可以是intset和hashtable，当满足以下两个条件的时候才会采用intset编码：

  1. 集合对象保存的所有元素都哦是整数值
  2. 集合对象保存的元素数量不超过512个

使用hashtable时，字典的每个键都是一个字符串对象，每个字符串对象包含了一个集合元素，而字典的值全部被设置为null。

有序集合对象(zset)

有序集合对象的编码可以是ziplist和skiplist。使用ziplist时，第一个节点保存元素的成员，第二个节点保存元素的分值。而使用skiplist编码的时候，会同时创建一个跳表和一个字典，跳表用来做ZRANK, ZRANGE等范围型操作，字典的键保存了元素的成员值保存了元素的分值，可以用O(1)查找元素的分值。当满足以下两个条件的时候才会采用ziplist编码：

  1. 有序集合保存的元素数量小于128个
  2. 有序集合保存的所有元素成员的长度都小于64字节
内存回收

引用计数法，使用redisObject中的refcount。然而refcount除了引用计数之外还有对象共享的作用。Redis在初始化的时候会创建一万个字符串对象，包含从0-9999的所有整数值，当需要用到的时候直接引用，refcount+1。Redis只对包含整数值的字符串对象进行共享。

单机数据库

Redis初始默认会创建16个db。

设置键的过期时间

EXPIRE(剩余秒数), PEXPIRE(剩余毫秒数), EXPIREAT(指定秒级时间戳), PEXPIREAT(指定毫秒级时间戳)。这四种设置过期时间最后都是通过PEXPIREAT来实现的。

过期键删除策略有以下三种：

  1. 定时删除
  2. 惰性删除
  3. 定期删除

Redis采用的时惰性删除和定期删除两种策略结合。定期删除是每当Redis的服务器周期性的执行serverCron函数时，会在规定的时间内从数据库的expires字典中随机检查一部分键的过期时间，并删除其中的过期键。

内存回收策略

假如 Redis 当中所有的键都没有过期，而且此时内存满了，就需要执行不同的淘汰策略来处理这种场景。

  1. volatile-lru，allkeys-lru

    基于抽样的lru，随机抽取maxmemory_samples个样本，然后通过lru删除。redisObject 对象中存在一个 lru 属性

  2. volatile-lfu，allkeys-lfu

  3. volatile-random，allkeys-random

  4. volatile-ttl

  5. noeviction

RDB

RDB将某个时间点上的数据库状态保存到一个RDB文件中，RDB文件是一个经过压缩的二进制文件。通过SAVE和BGSAVE两个命令来实现，其中SAVE会阻塞Redis服务器进程直到RDB文件创建完成；而BGSAVE会派生一个子进程，由子进程负责创建RDB文件，父进程继续处理请求。

通过使用save配置，来定义BGSAVE的执行条件。例如save 900 1，表示900秒内至少有1次数据库的修改。serverCron每隔100毫秒执行一次，会检查saveparams中的条件是否达到，达到了就执行BGSAVE。

AOF

如果开启了AOF，则服务器在执行完一个写命令后，会将写命令追加到服务器的aof_buf的末尾。

flushAppendOnlyFile有三种不同的flush aof缓冲区方式：

  1. always，将AOF缓存区内容写入AOF文件，并同步（fsync）
  2. everysec，如果上次AOF文件（页缓存）同步的事件超过一秒，则同步

事件

事件包括文件事件和时间事件两类。服务器与客户端的通信会产生文件事件，时间事件比如serverCron等。

serverCron平均100毫秒执行一次，主要工作包括：

  • 更新服务器的各类统计信息
  • 清除过期键
  • 关闭和清理连接失效的客户端
  • 尝试进行RDB或AOF
  • 如果是主服务器，对从服务器执行定期同步
  • 如果处于集群模式，对集群进行定期同步和连接测试
