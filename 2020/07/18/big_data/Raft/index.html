lab 2b

raft保证了一下属性来保证其一致性

  1. Election Safety：一个term只有一个leader

  2. Leader Append-Only：leader的log不会删除或覆盖，只会添加写

  3. Log Matching：相同index和term的两个log，那么这两个server该log和其之前的log全都相同

    1. 相同index和term的两个log，他们存有相同的commend（他们相同）

      If two entries in different logs have the same index and term, then they store the same command # fig3

    2. 相同index和term的两个log，他们之前的logs全都相同

      If two entries in different logs have the same index and term, then the logs are identical in all preceding entries # fig3

  4. Leader Completeness：如果一个log在某个term被提交了(committed)，那么之后的leader（term更高的leader）的logs里面都会有该log

  5. State Machine Safety：如果server apply某一index上的log，那么其他server不会在相同index apply不同的log

其他

此外还有很多恼人的小bug，多半是编码出错或论文理解不清楚。这里mark一下论文中一些参数的作用。

  1. nextIndex数组

    记录了每个server下一个log应该追加的位置

  2. matchIndex数组

    每个server当前log数组中最后一个log的index(len(log) - 1)

  3. lastApplied

    记录了上次apply到自己的state mechine的log的index

  4. commitIndex

    最高的已经被commit的log的index。leader会检查一个log是否被复制到了大多数的server，如果是，则设置commit = 该log的Index。该操作通过检查matchIndex，从(len(log) - 1)即最后一个log往前数，如果大多数peer的matchIndex都大于该一个index，则认为该index的log及其之前的log都被复制到了大多数的server # Log Matching。更新了commitIndex，就得检查lastApplied是否小于commitIndex，如果是，则表示把已经commit却没有apply到state mechine的log apply一下。具体的apply操作就是往applyCh里面写数据

  5. applyCh

    当要apply log到本地的state machine时，往该chan写数据

  6. PrevLogIndex

    AppendEntries RPC传递的参数，leader每次向follower append entries的时候，prevLogIndex = nextIndex[peer] - 1，表示要添加的index前一个index的log，即peer的最后一个log是否和leader匹配，如果不匹配，再根据 # fig 2做相关操作。

    1. 如果我(follower)的最后一个Log都不是prevLogIndex, 即leader的prevLogIndex超出了我的最大logIndex 或 在prevLogIndex这个位置我和leader的log的term不一致，则 # fig 2 AppendEntries RPC # 2，doesn’t match，返回错误
    2. 至少在prevLogIndex对上了，然后就检查传过来的logs和我本地prevLogIndex之后的log有没有不同，如果这段log发现有不同，就删除该不同的log和之后的logs，然后把传过来的logs加上。 # fig 2 AppendEntries RPC # 3 4

lab3B

3B思考

3B要做的事是实现raft的日志压缩功能。大概分以下几个步骤：

  1. kv server检测raft的日志大小，当日志大小达到maxraftstate的时候就压缩本地raft的日志
  2. 如果 raft follower 发现自己log存在conflict，然后来leader这里取log的时候发现leader的log已经被压缩了，这时leader就要发送InstallSnapshot RPC。

对于1这点，kv server在什么时候检测呢，单独再开一个go routine检测肯定不划算，go routine的检测间隔也不好控制。因此，可以在log增加的时候检测，即raft将某一条log apply的时候，在kv server这边就是apply chan里面又读到内容的时候。

对于2，leader什么时候发送InstallSnapshot RPC呢——如果发现 prevLogIndex < LastIncludeIndex 的时候，就表示follower需要的log已经被做成快照了，这时候就需要leader发送InstallSnapshot RPC了。这里需要注意的是发送InstallSnapshot RPC的是raft模块，收到该RPC的raft完成snapshot之后还需要通知其kv server更新 db。

3B snapshot 具体步骤如下：

  1. KVServer收到Apply时去检测当前RaftStateSize(RaftState包含currentTerm、voteFor、LogEntries、LastIndudeTerm、LastIndudeIndex)，如果大于阈值则出发KVServer doSnapshot
  2. KVServer将自己的DB和clientLastSeq Encode之后和lastApplyIndex一起发送给Raft
  3. Raft收到之后根据lastApplyIndex阶段logEntries，并更新LastIncludedIndex和LastIncludedTerm
  4. Raft将RaftState和KVServer传过来的数据一起保存

上述是Leader会干的事，Leader做完Snapshot之后，Follower的nextIndex可能在LogEntries里找不到了，这时触发InstallSnapshot RPC

  1. 使用InstallSnapshot RPC来代替AppendEntries RPC
  2. Follower收到InstallSnapshot RPC之后根据情况更改RaftState，并通过Chan Apply到Follower的KVServer
  3. KVServer读到数据后更新自己的DB和clientLastSeq

思考

如果有3个server[1, 2, 3]。开始1为leader，3个节点的Log都是up-to-date，所有节点都能正常通信。然后，1和3之间不能通信了。这是如果在一段时间内都没有Start操作，即没有新的Log进来，那么由于1，3的log都是新的，他们都可以当选leader。因此就会在1和3之间一直摇摆leader。

在第6张最后有一个解决方案，如果raft确信现在有leader存在的情况下，他会直接丢弃所有RequestVote RPC。怎么确信有leader存在呢，论文里说到的是，从收到leader的心跳到最低ElectionTimeout这段时间都是确信有leader的，即在这个区间收到RequestVote RPC都直接丢弃。

如果某一个follower一直连不上，那么他会一直增大自己的term参与竞选，导致其连上的时候term非常大，扰乱正常
